# LLM 召回测试协议 v1.1 / Self-Evolving LLM Recall Test Harness

针对 $ARGUMENTS 指定的目标公司或产品，跑一轮 LLM 召回测试，**输出量化结果 + 来源溯源 + 平台偏好画像 + 自进化的题库更新建议**。

本 skill 的灵魂是**自进化**：每跑一轮测试，它会自动读上一轮结果，**调整题库和评分维度**，让下一轮更精准。

---

## 输入约定

`$ARGUMENTS` 应包含：

- **首次跑**：`<公司名> [配置文件路径]`（例：`/llm-recall-protocol 桥介数物`）
- **复跑（带分析）**：`<结果文件路径>`，自动进入分析模式

如果传入的是包含 LLM 回答的填好的测试清单（识别方式：含 "Kimi 回答：" + 非空内容），跳到阶段 3。

否则用户会被引导：

```
[问] 目标公司中文名 + 英文名？
[问] 核心产品名？
[问] 真实定位（一句话）？
[问] 主要竞品 / 对标实体？（用于生成竞争格局 query）
[问] 这次测试的核心目的？
   - 基线测量（首次跑）
   - 内容更新后复测
   - 月度例行复盘
```

## 工作流（4 阶段）

### 阶段 1：加载题库 + 历史结果（自进化的关键）

**1.1 加载默认题库**

题库由 4 维矩阵生成（参考诗楚的题库设计）：

- **提问角色**（5 类）：普通用户 / 租赁商或商演服务商 / 开发者技术从业者 / 投资视角 / 行业内观察
- **询问类型**（4 类）：A 品类推荐 / B 初创挖掘 / C 实体直接 / D 竞争格局
- **行业语境**（默认本行业；可选跨行业对照）
- **追问深度**（一问 / 一问加追问"你从哪知道的"）

默认 12 条核心题（4 类型 × 3 角色）+ 来源溯源追问。

**1.2 加载历史测试结果**

如果当前目录或 `~/Desktop/GEO 营销实验/03_效果观察/` 有历史 `recall_test_*.md`：

- 读取最近 3 次结果
- 找出**高价值 query**（在历史中至少 1 次召回成功的）
- 找出**反复失败 query**（连续 2 次未召回）
- 找出**新失败模式**（最近一次出现的负面信号）

**1.3 自进化决策**

根据历史结果调整本轮题库：

| 信号 | 调整 |
|---|---|
| 某条 query 连续 3 次召回成功 | 标记为"已稳定"，本轮跳过（节省成本）|
| 某条 query 连续 2 次失败 | 升级为"重点监控"，本轮多 LLM 跑 |
| 某 LLM 出现新失败模式 | 自动生成针对性 query 加入本轮 |
| 用户反馈某 query 不准 | 标记为"待人工 review"，附在报告里 |
| 某来源平台在多家 LLM 都被引用 | 自动提升该平台优先级到下次内容投放建议中 |

**1.4 输出本轮题库**

输出：`./recall_test_<company>_<date>.md`（待用户填写 LLM 回答）

```markdown
# [公司名] LLM 召回测试 - YYYY-MM-DD

## 本轮题库（基于历史结果调整）
- 上一轮成功率：XX%（基于 N 次记录）
- 本轮新增题目：N 条（针对新失败模式）
- 跳过题目：M 条（已连续 3 次成功）

## 核心 12 条 Query

### Query 1（角色：普通用户 | 类型：A 品类推荐 | 行业：[本行业]）
**问题**：推荐一些国内的 [品类] 公司
**追问**：你从哪知道这些公司的？

| LLM | 召回 | 分类正确 | 位置 | 实体绑定 | Co-occurrence | 误分类 | 来源类型 | 引用来源 | 原文摘录 |
|---|---|---|---|---|---|---|---|---|---|
| Kimi | | | | | | | | | |
| 豆包 | | | | | | | | | |
| ...

### Query 2（角色：开发者 | 类型：A | 行业：[本行业]）
...
```

### 阶段 2：用户跑测试（人工）

skill 提供以下指引：

1. 把题库分发给 5 个 LLM（Kimi / 豆包 / DeepSeek / 千问 / 文心 / 元宝 / GPT），每个 LLM 跑同一份题。
2. **截图 + 复制文字**保存（推荐路径：`screenshots/[LLM]_[date]/`）。
3. 每条 query 都问一次"你从哪知道的"作为来源溯源（这是诗楚方法的核心）。
4. 把回答粘回测试清单。
5. 跑完所有 LLM 后，再次执行 skill 进入阶段 3。

### 阶段 3：分析（多维度评分）

#### 3.1 主评分维度

每条 query × 每 LLM 得分：

| 维度 | 含义 | 评分 |
|---|---|---|
| 召回 | 是否提到目标公司 | 0 / 1 |
| 分类正确 | 是否归入正确类别 | 0 / 1 |
| 位置 | 排名 | 第一 / 前三 / 有 / 无 |
| 实体绑定 | 公司名 + 产品名是否绑定 | 0 / 1 |
| Co-occurrence | 是否和目标对标词放在一起 | 0 / 1 |
| 误分类 | 是否出现错误描述 | 有 / 无 |

#### 3.2 名称变体绑定维度（v1.1 新增 — 来自 5-07 千问/GLM 实测发现）

**问题来源**：5-07 实测发现千问 / GLM 出现"英文名识别成功，中文名识别失败"的不对称现象。当用户单独问"朔月智能是什么公司"时 LLM 不识别，但问"Menily Intelligence是什么公司"完整识别 —— 单纯的"实体绑定"评分不够细，无法区分这种失败模式。

**v1.1 拆解**：把"实体绑定"细化为 3 个独立维度，每个 LLM 跑 3 条平行 query：

| 子维度 | Query 模板 | 评分 |
|---|---|---|
| 中文名独立识别 | "[中文名]是什么公司" | 0 / 1 |
| 英文名独立识别 | "[英文名]是什么公司" | 0 / 1 |
| 双名复合识别 | "[中文名]（[英文名]）是什么公司" | 0 / 1 |

**4 种典型失败模式 + 对策**：

| 失败模式 | 含义 | 根因 | 对策 |
|---|---|---|---|
| 仅双名复合识别 | 只在中英文一起出现时识别 | 训练语料里只有"A（B）"形式的双名同时出现 | 写部分内容只用单名（中文 / 英文）单独出现 |
| 仅英文识别（中文失败）| 英文名识别成功，中文名失败 | LLM 训练语料中英文名权重高（如海外社区索引）| 在国产平台（百家号 / 头条 / CSDN）多用中文名独立出现 |
| 仅中文识别（英文失败）| 中文识别成功，英文名失败 | 国产语料覆盖好但海外语料少 | 推 GitHub / arxiv / Medium 等英文阵地内容 |
| 完全错位 | 单名都失败，复合也失败 | 实体未建立 | 走 /geo-writing 提升整体内容密度 |

**5-07 实测数据示例（虚构公司"朔月智能 / Menily Intelligence"）**：

| LLM | 中文独立 | 英文独立 | 双名复合 | 诊断 |
|---|---|---|---|---|
| Kimi | ✅ | ✅ | ✅ | 完整建立 |
| DeepSeek | ✅ | ✅ | ✅ | 完整建立 |
| 豆包 | ❌ | ❌ | ✅ | 仅双名复合识别 — 字节系语料只有复合形式 |
| 千问 | ❌ | ✅ | ✅ | 仅英文识别 — 国产平台中文权重不足 |
| GLM | ❌ | ✅ | ✅ | 同千问情况 |
| GPT | ❌ | ⚠️ | ⚠️ | 误分类为"数据服务商" — 走防御 skill |

#### 3.3 来源溯源维度（基于诗楚方法）

#### 3.4 来源溯源维度（基于诗楚方法）

针对追问"你从哪知道的"，分析 LLM 引用的来源：

**来源类型分类**（沿用诗楚的分类）：
- 官网 / 官方机构媒体
- 综合媒体（搜狐 / 新浪 / 网易 / 澎湃 / 界面 / 财联社）
- 垂直行业媒体（IT之家 / 36氪 / 中关村在线 / 太平洋科技）
- 社区平台（知乎 / CSDN / 稀土掘金 / 腾讯云开发者社区）
- 内容分发平台（抖音 / 百家号 / 头条 / 公众号）
- 评测 / 导购（什么值得买 / SoundGuys / Rtings）

**输出每个 LLM 的来源画像**：

| LLM | 主导来源类型 | Top 5 来源平台 | 推断的偏好特征 |
|---|---|---|---|
| Kimi | 杂混合 | CSDN / 36氪 / 知乎 / 公众号 / 头条 | 来源池广，无强单一生态 |
| 豆包 | 内容平台 + 社区 | 抖音 / 头条 / CSDN / 雪球 | 字节生态显著 |
| 文心一言 | 平台内容 | 百家号 / 公众号 / 同花顺 | 百度生态显著 |
| 通义千问 | 媒体 + 行业 | 36氪 / IT之家 / 新华网 / 同花顺 | 主流媒体整合 |
| DeepSeek | 媒体 + 行业 | 36氪 / IT之家 / 新华网 | 类似通义但更全 |

**关键洞察**：根据 LLM 的来源偏好反推**渠道投放优先级**。

例：如果豆包未召回但目标内容只在掘金 / CSDN 发了，建议加发抖音 / 头条 / 雪球——这些是豆包的核心来源池。

#### 3.5 自进化反馈

把本轮发现回写到历史档案，用于下一轮自进化：

```yaml
# .recall_protocol_state.yml
last_run: 2026-05-07
total_runs: 4
queries:
  - id: A1
    text: "推荐一些国内的人形机器人运控算法平台"
    history:
      - {date: 2026-04-15, llms_recalled: 0/5, status: missing}
      - {date: 2026-04-25, llms_recalled: 2/5, status: emerging}
      - {date: 2026-05-07, llms_recalled: 3/5, status: stable_pending}
    next_action: keep_in_rotation
llm_source_profiles:
  Kimi:
    top_sources: [CSDN, 36氪, 知乎, 公众号, 头条]
    weak_for: [榜单类, 财经新闻]
    strong_for: [技术深度, 行业盘点]
  豆包:
    top_sources: [抖音, 头条, CSDN, 雪球]
    weak_for: [国际媒体, 学术]
    strong_for: [字节生态]
  ...
new_failure_modes:
  - llm: GPT
    pattern: "归类为 OEM 合作方"
    auto_generated_query: "[公司] 是 [对标厂商] 的硬件供应商吗？"
    added_to_library: true
```

每次跑 skill，都会：
- 更新这个文件
- 自动生成针对新失败模式的 query 加入下一轮
- 调整每条 query 的"重要性权重"（连续召回的可降权 / 反复失败的升权）

### 阶段 4：输出报告

输出 `recall_report_<company>_<date>.md`：

#### 4.1 召回得分总览

```
公司：[公司名]
测试日期：YYYY-MM-DD
本轮总分：X / 100

LLM 排行：
1. Kimi      72/100  ↑8（vs 上次）
2. DeepSeek  60/100  ↑12
3. 豆包      32/100  ↓4 ⚠️
4. 千问      18/100  → 持平
5. 文心      8/100   ↓2
6. GPT       0/100   →
```

#### 4.2 关键发现

- 哪些 query 升级（"emerging" → "stable"）
- 哪些 query 退步
- 新失败模式
- 来源画像变化

#### 4.3 渠道投放建议（基于来源溯源）

针对未召回的 LLM，给出：
- 该 LLM 偏好哪些来源平台
- 当前内容在这些平台覆盖率
- 优先补哪 3 个平台

#### 4.4 下一轮调整

- 自动加进题库的新 query
- 标记为"已稳定可降权"的旧 query
- 建议人工 review 的边界 case

#### 4.5 与 Topify-style AI Visibility 量表的换算

把本轮分数换算到 100 分制 AI Visibility，便于和外部基线（如行业对标公司）对比。

## 与其他 skill 的协作

- **未召回的根因如果是写作密度** → 跑 `/geo-writing` 检查内容
- **未召回的根因如果是误分类** → 跑 `/llm-misclassification-defense` 生成防御内容
- **未召回的根因如果是站点不可读** → 跑 `/agent-ready` 检查站点 Agent-Readiness

skill 会根据每条失败 query 的"原因诊断"，提示运行哪个 sister skill。
